[!NOTE] Definition
Assume a database with a set of tuples; each tuple consists of a unique fixed-length key together with some state information. A query specifies a key K. The goal is to return the state information associated with the tuple whose key is K.

routers = interconnect devices

Why do we learn Exact Match?

  • In the context of networking
    • We measure the speed in terms of memory accesses, instead of processing
    • We have a strict deadline (under 10 ns)
  • Crucial for bridging, which is often integrated within a router

Bridging

  • 3 challenges: Ethernet, Wire Speed Forwarding, and Scaling Lookups to Higher Speeds
    • 802.1 spanning tree bridges

Challenge 1: Ethernet under fire

  • Problems: poor performance at large loads (retransmission) and long distance
  • Routers interconnect links using information in the routing header, while repeaters interconnect links based on physical-layer information, such as bits.
  • Bridge (Data Link Layer Relay)
    • based only on the data link layer
    • Packet Repeater
      • Buffers packet, even for packets that don’t cross different Ethernets
    • Filtered Repeater
      • has a table that maps station addresses to Ethernets (A and B are in the same network)
      • Define p: fraction of traffic on each Ethernet is to destinations on the same Ethernet. Then the overall bandwidth of the two Ethernet systems becomes (1 + p) times the bandwidth of a single Ethernet.
      • This follows because the fraction p can be simultaneously sent on both Ethernets, increasing overall bandwidth by this fraction. Hence both bandwidth and distance increase.
    • Filtering Repeater With Learning

challenge 2: wire speed forwarding

  • If the bridge takes more time to forward a packet than the minimum packet arrival time, there are always scenarios in which packets to be forwarded will be dropped because the buffers are filled with packets that will be discarded.
  • The solution: the bridge would look up the destination address in the table (for forwarding) and the source address (for learning) in the time it took a minimum-size packet to arrive on an Ethernet
    • 64-byte minimum packet: 51.2 microsecond to forward a packet.
    • Two-port bridge could receive a minimum-size packet on each of its Ethernets every 51.2 microsecond: two lookups (destination and source) every 25.6 microsecond.
    • The new design will process each packet, determining whether or not to discard the packet
    • use binary search lookups

Challenge 3: scaling lookups to higher speeds

  • Bridge database size increases to 64K. Binary search lookups don’t suffice
  • Gigaswitch: hashing-based lookups. Each bridge lookup makes at most four reads from memory.

Perfect Hashing

  • hashing
    • key: ethernet address
    • value: port number?
  • A hashed address is obtained by the equation A(X) ∗ M(X) mod G(X)
    • A(X): address expressed as a 47-degree polynomial
    • M(X): nonzero, 47-degree programmable hash multiplier
    • G(X): irreducible polynomial X48+X36+X25+X10+1X^{48} + X^{36} + X^{25} + X^{10} + 1X48+X36+X25+X10+1
  • Hashed Address
    • 48-bit: size of Ethernet address
    • [15:0]: index into a 64K-entry hash table
    • Each entry points to a balanced binary tree of height at most 3
    • [47:16]: distinguish collided keys
      • The binary tree is sorted based on these values
    • at most 4 memory accesses: 1 for hash table lookup, 3 for the height-3 tree
  • CAM (Content Addressable Memory): a small hardware lookup database. Aimed to deal with 7 conflicts (size of the heigh-3 complete binary tree)

Using hardware parallelism

  • pipeline binary search
  • logarithmic number of processing stages, each with its own memory array
    • Each level of the binary tree has its own stage
  • Search keys enter from the left and progress from stage to stage, carrying a pointer (points to a “child” key) that identifies which key in the corresponding stage memory must be compared to the search key.
  • The lookup throughput is nearly one per memory access
  • still log⁡2n\log_{2}nlog2​n lookups in total, but the throughput is improved

d-left

  • when n objects are hashed into n buckets, the expectation of this maximum is (1+o(1))log⁡nlog⁡log⁡n\frac{(1 + o(1))\log n}{\log\log n}loglogn(1+o(1))logn​ (high probability) when a random hash function is used
  • Goal: reduce this maximum to a small number (say 2 or 3) so that every bucket can fit into a memory block
  • d-random
    • use d > 1 random hash functions (h1, h2, ···, hd) instead of 1. Given a hash key x, there are d candidate hash buckets into which x can be inserted: buckets indexed by h1(x), h2(x), ···, hd(x) respectively.
      • x is inserted into the bucket that is least occupied
    • The expectation of the maximum value is (log⁡log⁡nlog⁡d)+O(1)\left( \frac{\log \log n}{\log d} \right) + O(1)(logdloglogn​)+O(1)
  • d-left
    • variation of d-random
    • Hash table is partitioned into logically independent subsets of size nd\frac{n}{d}dn​. d-random doesn’t have such partition
    • if there are more than one least occupied buckets among the d buckets where an object x should be inserted into, then x will be inserted into the leftmost one among them
