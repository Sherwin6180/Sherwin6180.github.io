History of DBMS continued

2000s: Rise of NoSQL Databases

  • Characteristics:

    • Schemaless: Flexible data models without rigid predefined schemas.
    • No ACID Transactions: Focused on eventual consistency rather than strict ACID properties.
    • Designed For: High availability, high scalability, and distributed environments.
    • APIs: Utilize custom, often simple, APIs instead of standard SQL.

  • Examples:

    • MongoDB, Cassandra, CouchDB, Redis: Popular systems each with unique data models (document, wide-column, key-value, etc.).
    • Redis Exception: Although originally non-relational and schemaless, Redis has evolved to support some ACID-like features and can now interface with SQL-like queries through extensions.

2010s: Diversification and Hybrid Systems

During this decade, several new types emerged, blending features of traditional RDBMS with modern distributed and scalable architectures.

  1. NewSQL Databases:

    • Goal: Combine the consistency and transaction support of relational databases (ACID, SQL) with the scalability of NoSQL systems.
    • Technologies:
      • Google Spanner: A globally-distributed NewSQL database that offers strong consistency, SQL support, and horizontal scalability using advanced clock synchronization (TrueTime API).
      • Other examples include CockroachDB and TiDB, which aim for similar goals.

  2. Cloud Systems (DBaaS):

    • Early DBaaS: Containerized versions of existing databases for deployment ease.
    • Cloud-Native DBaaS:
      • Designed from scratch for the cloud environment.
      • Focus on elasticity, managed services, and pay-as-you-go models.
      • Snowflake: A cloud-native data warehouse built for scalability, separation of compute and storage, and ease of integration with data lakes.
      • Other offerings include Amazon RDS/Aurora, Google Cloud Spanner, Azure SQL Database, etc.

  3. Shared Disk Engines / Data Lakes:

    • Architecture:
      • Separation of compute and storage layers.
      • Shared storage layer allows independent scaling of compute resources.
    • Use Case: Large-scale analytics, storing diverse data types from various sources.
    • Snowflake Note: Snowflake uses a similar architecture, separating compute clusters from a central storage layer to serve multiple concurrent queries efficiently.
    • Often associated with data lake architectures that aggregate raw data for processing and analysis.

  4. Graph Systems:

    • Purpose: Specialized for storing and querying graph-structured data (networks, relationships).
    • Features:
      • Graph-centric Query APIs: Languages like Cypher (Neo4j) or Gremlin (Apache TinkerPop).
      • Emphasize traversals, pattern matching, and relationship-oriented queries.
    • Relation to Historical Models: Similar in spirit to the network data model (e.g., CODASYL), but with modern abstractions and optimizations.
    • Examples: Neo4j, Amazon Neptune, JanusGraph.

  5. Time Series Systems:

    • Specialization: Designed to efficiently handle time-stamped (temporal) data, such as metrics, events, IoT sensor data.
    • Key Features:
      • Optimized for high ingestion rates, retention policies, time-based queries, and aggregations.
    • Examples:
      • InfluxDB: A popular time series database.
      • Others: TimescaleDB (built on PostgreSQL), Prometheus (for monitoring), etc.

2020s: Further Specialization and Innovation

  • Emerging Trends:

    • Embedded DBMS: Lightweight databases that run within applications (e.g., SQLite, Realm).
    • Multi-Model DBMS: Systems supporting multiple data models (e.g., document, graph, key-value) under one engine.
    • Hardware Acceleration: Use of specialized hardware (GPUs, FPGAs, NVMe drives) to accelerate database operations.
    • Blockchain DBMS: Decentralized, tamper-resistant ledgers that blend blockchain technology with database features.

  • Convergence of SQL and NoSQL:

    • Modern NoSQL databases increasingly offer SQL interfaces or SQL-like query languages.
    • This trend reduces friction for users familiar with SQL while maintaining the benefits of NoSQL systems (scalability, flexibility).

Corrections & Clarifications

  • Google Spanner:
    • While initially mentioned as a NoSQL example, it is better categorized under NewSQL because it provides strong consistency, SQL support, and ACID transactions in a distributed setting.
  • Redis Specificity:
    • Redis, once a pure in-memory NoSQL key-value store without schema or ACID, has evolved. Many modern NoSQL systems, unlike Redis, now support SQL or SQL-like queries, but Redis remains somewhat unique with its specific feature set (e.g., in-memory data structures, optional persistence).

Data Model & Relational Algebra

Data Models

  • Definition: A data model defines how data is structured, manipulated, and constrained in a database.

  • Examples:

    • Relational: Used in RDBMS.
    • Key-Value, Graph, Document, Column Family: Common in NoSQL systems.
    • Array/Matrix: Often used in machine learning and scientific applications.
    • Hierarchical and Network: Legacy models, now mostly obsolete.

  • Three Core Aspects of Data Models:

    1. Structure of Data: Conceptual design, such as schemas and relations.
    2. Operations on Data: Methods for querying and manipulating data (e.g., relational algebra).
    3. Constraints on Data: Rules ensuring data validity (e.g., domain constraints).

Structure of Data

  1. Relational Schema:

    • Example:

      Students(SID:String,Name:String,GPA:Float)Students(SID: String, Name: String, GPA: Float)Students(SID:String,Name:String,GPA:Float)

      • Relation Name: Students.
      • Attributes: SID, Name, GPA.
      • Domains: Data types for attributes (String, Float).

    • Relation vs. Schema:

      • Relation: A set of tuples (rows) representing the data.
      • Schema: A set of attributes (columns) defining the structure.
        • Schema describes what the data should look like.
        • Relation contains the actual data.

  2. Relational Data:

    • Key Terms:
      • Row/Tuple/Record: Interchangeable terms for a single data entry.
      • Column/Attribute: Represents a specific property or field in the schema.
      • Cardinality: The number of rows in a table.
      • Arity: The number of attributes (columns) in a schema.
    • Relational Instance: A specific set of tuples conforming to the schema.
      • Instance is a multi-set (allows duplicates) rather than a strict set.
    • Order Independence:
      • In relational models, the order of rows and columns does not matter.
      • For example, a table with 4 rows and 3 columns can be represented in 4!×3!4! \times 3!4!×3! ways.

  3. Set Representation:

    • A relation instance is a subset of the Cartesian product of the domains of its attributes:

      R⊆Domain1×Domain2×⋯×DomainnR \subseteq \text{Domain}_1 \times \text{Domain}_2 \times \dots \times \text{Domain}_n R⊆Domain1​×Domain2​×⋯×Domainn​

    • A relational database instance is a collection of relation instances, satisfying all constraints.

  4. Constraints:

    • Domain Constraints: Ensure that attribute values fall within their defined domains.

Summary

  • Relation: A set of tuples (actual data).
  • Schema: A set of attributes (structure definition).
  • Relational Instance: A specific instance of a relation (conforms to the schema).
  • Relational Database Instance: The complete database, containing multiple relation instances.
  • Key Properties:
    • Relations are represented as sets (order of rows/columns doesn't matter).
    • Valid instances satisfy all domain constraints.

Operations on Data and Relational Algebra

Relational Algebra and SQL

  • SQL:

    • Declarative Query Language: Specifies what the user wants but not how to achieve it.
    • Example: SELECT name FROM Students WHERE GPA > 3.5;

  • Relational Algebra:

    • Acts as the intermediate language between SQL queries and their execution.
    • Translates declarative SQL queries into a sequence of operators that the system can execute.
    • Provides a formal foundation for query processing and optimization.

Relational Model vs. Key-Value Model

Aspect Relational Model Key-Value Model
Structure Relations (tables) Key-value pairs
Operations Relational algebra (e.g., selection, projection, join) GET, PUT, DELETE
Constraints Various constraints (listed below) Key must be unique; value must not be null
Flexibility Less flexible (data types and schema are defined) More flexible (can store arbitrary values)
Queryability More queryable (data is structured and relationships exist) Less queryable (data inside the key is opaque)
Performance Optimizable (knowledge of data types aids optimization) Limited optimization due to less data knowledge

Types of Constraints in the Relational Model

  1. Key Constraints:

    • Each relation must have a primary key (e.g., SID in Students).
    • Ensures uniqueness for each tuple.

  2. Domain Constraints:

    • Attribute values must conform to their specified domain (e.g., GPA must be a float).

  3. Referential Integrity Constraints:

    • Foreign keys must match primary keys in referenced tables.
    • Ensures consistency between related tables.

  4. Not Null Constraints:

    • Certain attributes cannot have null values (e.g., Name cannot be null).

  5. Unique Constraints:

    • Certain attributes must have unique values (e.g., Email in a Users table).

  6. Check Constraints:

    • Custom rules for values in attributes (e.g., GPA >= 0 AND GPA <= 4).

SQL Engine Workflow

  1. SQL Query (Declarative Query):

    • User specifies what to retrieve or update, not the execution process.
    • Example: SELECT GPA FROM Students WHERE Name = 'Alice';

  2. Relational Algebra Plan (RA Expression):

    • SQL query is translated into a sequence of relational algebra operations.
    • Example RA Expression for the query above:
      • Selection: Filter tuples where Name = 'Alice'.
      • Projection: Extract the GPA attribute.

  3. Optimized Relational Algebra Plan:

    • Logical Optimization:
      • Reorganize or rewrite the RA plan for better performance.
      • Techniques include:
        • Pushing down selections: Apply filters as early as possible.
        • Eliminating redundant operations: Remove unnecessary projections or joins.
    • Physical Optimization:
      • Convert logical plan into a physical plan by choosing specific algorithms for each operator (e.g., hash join vs. nested loop join).

  4. Execution Phase:

    • Each operator in the optimized plan is executed in sequence.
    • Execution strategies depend on:
      • Available indexes (e.g., B-trees, hash indexes).
      • Size of the data.
      • Query requirements (e.g., sorting or grouping).

Relational Algebra Operators

Basic Operators

  1. Selection (σ\sigmaσ)

    • Definition: Filters tuples from a relation based on a condition.
    • Symbol: σC(R)\sigma_{C}(R)σC​(R), where CCC is the condition and RRR is the relation.
    • Example:
      • Query: Select employees with a salary greater than 40,000.

        σsalary>40000(Employee)\sigma_{\text{salary} > 40000}(\text{Employee}) σsalary>40000​(Employee)

      • Condition Examples: =,<,≤,>,≥,≠=, <, \leq, >, \geq, \neq=,<,≤,>,≥,=.

  2. Projection (π\piπ)

    • Definition: Selects specific attributes (columns) from a relation and removes duplicates (as relational algebra works with sets).
    • Symbol: πA1,A2,...,An(R)\pi_{A1, A2, ..., An}(R)πA1,A2,...,An​(R), where A1,A2,...,AnA1, A2, ..., AnA1,A2,...,An are the attributes and RRR is the relation.
    • Example:
      • Query: Select distinct student names and GPAs from Students.

        πSName, GPA(Students)\pi_{\text{SName, GPA}}(\text{Students}) πSName, GPA​(Students)

      • Combined Example:
        • Query: Select names and GPAs of students with GPA > 3.5.

          πSName, GPA(σGPA>3.5(Students))\pi_{\text{SName, GPA}}(\sigma_{\text{GPA} > 3.5}(\text{Students})) πSName, GPA​(σGPA>3.5​(Students))

        • Equivalent to:

          σGPA>3.5(πSName, GPA(Students))\sigma_{\text{GPA} > 3.5}(\pi_{\text{SName, GPA}}(\text{Students})) σGPA>3.5​(πSName, GPA​(Students))

  3. Cartesian Product (×\times×)

    • Definition: Combines every tuple in R1R1R1 with every tuple in R2R2R2.
    • Symbol: R1×R2R1 \times R2R1×R2.
    • Result: If R1R1R1 has mmm tuples and R2R2R2 has nnn tuples, the result has m×nm \times nm×n tuples.
    • Example:
      • Query: Combine all tuples from Students and Courses.

        Students×Courses\text{Students} \times \text{Courses} Students×Courses

  4. Union (∪\cup∪)

    • Definition: Combines tuples from two relations, removing duplicates.
    • Example:
      • Query: Get all students from two campuses.

        Students1∪Students2\text{Students}_1 \cup \text{Students}_2 Students1​∪Students2​

  5. Set Difference (−-−)

    • Definition: Returns tuples in R1R1R1 but not in R2R2R2.
    • Example:
      • Query: Students enrolled in Math but not Science.

        StudentsMath−StudentsScience\text{Students}_\text{Math} - \text{Students}_\text{Science} StudentsMath​−StudentsScience​

Relational Algebra Query Example

Query:

Relational Algebra:

πgpa, address(σgpa>3.5(Students⋈People))\pi_{\text{gpa, address}} \left( \sigma_{\text{gpa} > 3.5 } \left( \text{Students} \bowtie \text{People} \right) \right) πgpa, address​(σgpa>3.5​(Students⋈People))

Steps:

  1. Perform a Cartesian product: Students×People\text{Students} \times \text{People}Students×People.
  2. Apply selection: σGPA>3.5∧s.Name=p.Name\sigma_{\text{GPA} > 3.5 \land \text{s.Name} = \text{p.Name}}σGPA>3.5∧s.Name=p.Name​.
  3. Apply projection: πGPA, Address\pi_{\text{GPA, Address}}πGPA, Address​.
  4. Remove duplicates (implicit in relational algebra).

Derived Operators

  1. Union (∪\cup∪):

    • Combines tuples from two relations, removing duplicates.
    • Relational Algebra: R1∪R2R1 \cup R2R1∪R2.

  2. Difference (−-−):

    • Returns tuples in R1R1R1 but not in R2R2R2.
    • Relational Algebra: R1−R2R1 - R2R1−R2.

  3. Intersection (∩\cap∩):

    • Returns tuples common to R1R1R1 and R2R2R2.
    • Derived From:

      R1∩R2=R1−(R1−R2)R1 \cap R2 = R1 - (R1 - R2) R1∩R2=R1−(R1−R2)

  4. Natural Join (⋈\bowtie⋈):

    • Joins two relations by matching values in all shared attributes.

    • R1⋈R2R_1 \bowtie R_2R1​⋈R2​: Joins R1R_1R1​ and R2R_2R2​ on equality of all shared attributes.

      • If R1R_1R1​ has attribute set AAA, and R2R_2R2​ has attribute set BBB, and they share attributes A∩B=CA \cap B = CA∩B=C, it can also be written as: R1⋈CR2R_1 \bowtie_C R_2R1​⋈C​R2​

    • Relational Algebra:

      R1⋈R2=πA∪B(σC=D(ρC→D(R1)×R2))R_1 \bowtie R_2 = \pi_{A \cup B} \left( \sigma_{C = D} \left( \rho_{C \to D}(R_1) \times R_2 \right) \right) R1​⋈R2​=πA∪B​(σC=D​(ρC→D​(R1​)×R2​))

    • This explanation provides details about the components of a relational algebra expression for a natural join (R1⋈R2R_1 \bowtie R_2R1​⋈R2​).

    • Breakdown:

      1. Rename (ρC→D\rho_{C \to D}ρC→D​):
        • Renames one of the shared attributes (CCC) in one of the relations (R1R_1R1​) to ensure there is no naming conflict during the Cartesian product (×\times×).
        • Purpose: Since natural join matches shared attributes, renaming avoids ambiguities by ensuring the attribute names in R1R_1R1​ and R2R_2R2​ are distinct.
      2. Selection (σC=D\sigma_{C=D}σC=D​):
        • After performing the Cartesian product (R1×R2R_1 \times R_2R1​×R2​), this operation filters the rows where the values of the renamed attribute (DDD) from R1R_1R1​ match the corresponding values of CCC in R2R_2R2​.
        • Purpose: Ensures that only rows satisfying the equality condition on shared attributes are included in the result.
      3. Projection (πA∪B\pi_{A \cup B}πA∪B​):
        • Projects only the attributes A∪BA \cup BA∪B, which include all attributes from both relations, but eliminates duplicate columns (attributes) resulting from the shared attribute.
        • Purpose: Produces the final result of the natural join, removing any duplicate shared attributes while retaining the distinct attribute values.

    • Example:

      • Query: Join Students and People on Name (using theta join):

        Students⋈S.Name=P.NamePeople\text{Students} \bowtie_{\text{S.Name} = \text{P.Name}} \text{People} Students⋈S.Name=P.Name​People

    • Degenerate Case: If R(A,B,C)R(A,B,C)R(A,B,C) and S(D,E)S(D,E)S(D,E) have no shared attributes, ⋈\bowtie⋈ degenerates to R×S=(A,B,C,D,E)R \times S=(A,B,C,D,E)R×S=(A,B,C,D,E).

  5. Theta Join (⋈Θ\bowtie_\Theta⋈Θ​):

    • Joins two relations based on a predicate Θ\ThetaΘ.
    • Relational Algebra:

      R1⋈ΘR2=σΘ(R1×R2)R1 \bowtie_\Theta R2 = \sigma_\Theta(R1 \times R2) R1⋈Θ​R2=σΘ​(R1×R2)

    • Example:
      • Query: Join Students and Courses where GPA > 3.5:

        σGPA>3.5(Students×Courses)\sigma_{\text{GPA} > 3.5}(\text{Students} \times \text{Courses}) σGPA>3.5​(Students×Courses)

Step 2: Cartesian Product (×\times×)

Perform the Cartesian product of ρS.name→C(S)\rho_{S.name \to C}(S)ρS.name→C​(S) and PPP (People P).

Filter the rows where C (from Students S) equals P.name (from People P).
