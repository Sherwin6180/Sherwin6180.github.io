接下来讨论运行时间性能，然后再聊速率。大家理解了吗？
好的，我们转向公式。

假设我写了一个程序，CPU时间公式：
时钟周期数 ÷ 时钟频率（以GHz为单位）。

无论需要多少周期运行程序，乘以时钟周期的时间，就可以得出程序的运行时间。这是最基本的逻辑。
不过，这还不能告诉我们太多信息，所以我们引入一个新概念：每指令平均周期数（CPI）。

CPI表示平均每个指令需要多少时钟周期。例如在一个5级流水线中，每一级流水线都在处理一条指令。就像组装线上造车，每个工位需要10秒，每10秒就有一辆车出厂。

于是我们可以写公式：
时钟周期数 = CPI × 指令数。

假设有100条指令，流水线每秒能完成1条指令，那么总共需要100个时钟周期。这是非常基础的公式之一，可以称之为“计算机架构中的牛顿定律”。

你可以用这个公式做一个纹身：
指令数 × 平均周期数 × 每周期时间 = 程序运行时间。

如何减少指令数量？

减少指令数量的方式是：即使你正在执行更复杂的指令，仍然保持总指令数不变，因为你可以通过复杂指令来分摊开销。

增加复杂的指令是个好主意。例如，我设计了一个新的指令，叫做“Excel表格”。

这是之前的一种趋势：不断增加指令的复杂度。一些微架构执行的指令集架构是经过简化的版本。当时看起来很有意义。我们稍后会更多地讨论这个问题。

我们还可以减少CPI（每指令平均周期数）。这意味着我们可以通过流水线或者并行性来提高每周期执行的指令数。但如果遇到分支怎么办？那就用预测技术（speculation），在这个方面我还有一些小技巧可以用。

接下来是减少周期时间（cycle time）。如何做到这一点呢？可以通过更深的流水线设计，把任务分成更小的阶段。例如，组装线上每个阶段需要10秒。如果我们把阶段分成5秒，工人只需要完成一半的任务，那么每辆车的总周期没有变化，但周期时间缩短了，也就提高了生产率。

此外，还可以通过技术缩放（technology scaling）来提高电路速度。不过，这些方法也会带来权衡。

例如： 如果我们减少指令数量，就需要更复杂的指令；增加CPI可能导致更复杂的流水线设计；减少周期时间需要更精细的分工。历史上，这些权衡因记忆体的限制而存在，例如早期的核心存储（core memory）昂贵且手工制作复杂。随着半导体存储的发明，这些限制被逐渐突破。

DARPA（美国国防高级研究计划局） 在80年代推出了一些硬件项目。为了吸引资金和关注，项目通常需要一个吸引人的名字。例如，当时一个来自伯克利的团队提出了“精简指令集计算机”（RISC, Reduced Instruction Set Computer）的概念。这个名字来源于他们在提案中反复提到“降低风险”（reduce risk）。相对应的复杂指令集计算机（CISC）听起来则像一种需要去医院治疗的病。

在RISC的发展过程中，他们的目标是设计规则简单的芯片。这种方法让研究生在课堂上能够设计一个包含30多条指令的处理器。由于功能有限，他们将某些“缺陷”设计成了“特性”，比如简单化的指令集。

与此同时，编译器的发展也经历了一个漫长的过程。从早期需要手动编写解析器到后来使用“编译器生成器”自动完成编译过程，这大大缩小了高级语言与指令集之间的语义差距。这也推动了计算机架构的进步。

他们认为通过直接在硬件中执行高级语言，可以减少语义差距并让编译器更容易实现。当然，真正执行高级语言的其实是微码（microcode）。编译器会将代码转化为一种解释器，然后通过微码来运行。这在当时非常流行，不仅仅因为编译器难以构建，还因为当时的内存非常昂贵。

但到了80年代，内存变得便宜了，编译器也变得更好。
在IBM的一个学生研究实验室中，有一个名叫John Cock的人被分配了一个任务。他被要求开发一种通道处理器（channel processor）。这是什么呢？IBM的大型计算机不希望浪费时间处理I/O操作，所以他们开发了可编程的DNA引擎，即通道处理器，用来处理数据传输。John Cock接下了这个任务，但他其实是个编译器专家，于是他决定利用这个项目尝试构建一个新的微架构。

复杂指令的难点

他注意到，当时的指令集（例如IBM 390和Digital Equipment Corporation的VAX）中，有很多复杂的指令，甚至包括一个能计算双精度多项式的指令。问题在于，编译器无法识别代码中需要使用这些复杂指令的场景，因此这些指令并不实用。而且，这些复杂指令的解码器非常大，拖慢了整个处理器的速度。

于是，John Cock决定创建一个新的指令集，只包括编译器真正会用到的指令。他将这个项目命名为“801项目”（Project 801），因为他的实验室位于Austin的801号大楼。这个项目的目标是简化指令集，让它们与编译器更匹配。

RISC（精简指令集计算机）的起源

大约在同一时间，斯坦福的一名助理教授（他的博士学位来自纽约州立大学石溪分校）与伯克利的一名研究人员合作。他们造访了John Cock，并提出了“RISC”（Reduced Instruction Set Computer）的概念。这个名字的灵感来自DARPA（美国国防高级研究计划局）的提案中反复提到的“降低风险”（reduce risk）。他们认为简化指令集不仅有助于硬件设计，还能更好地支持编译器优化。

关于ARM

有人知道ARM是什么意思吗？最早是“Acorn RISC Machines”，后来改成“Advanced RISC Machines”，但现在ARM官方网站甚至不再提及“RISC”。ARM的指令集在版本8中包含了数千条指令，从任何角度来看都算不上“精简”。此外，ARM的指令宽度并不固定，有标准指令、Thumb指令，以及介于两者之间的混合指令。

RISC-V现状

如果你对RISC-V感兴趣，可以去RISC-V委员会的网站下载它的指令集架构，数一数里面有多少条指令。结果会让你明白，设计处理器不能由委员会决定，因为总会有人提出额外的要求，比如加入离散余弦变换（DCT）指令，仅仅是为了安抚某个提出建议的人。

复杂指令的回归

早期（80年代末到90年代初），RISC的支持者认为复杂指令集计算机（CISC）的时代即将结束。然而，Intel和AMD找到了解决办法：他们通过解码器将复杂指令转化为简单指令，再在处理器中使用这些简单指令来执行。这种方法不仅保留了复杂指令集，还利用了解码器的灵活性来提升性能。通过这种方式，CISC并未消亡，反而继续发展。

结论

编译器技术在90年代之后取得了巨大进步，如今已经能够识别并使用复杂指令，例如向量指令。复杂指令的使用能够显著提高代码的运行效率，因此围绕RISC与CISC的争论已不再有意义。

举个例子，当Intel开发P6处理器（代号Pentium Pro）时，他们通过引入微操作（micro-ops）和乱序执行流水线，大大提升了处理器性能。这一创新实际上结束了RISC与CISC的争论。

假设程序1的吞吐量是4事务/秒，程序2在A上的吞吐量是7事务/秒。在B上，程序1的吞吐量是2事务/秒，程序2是7事务/秒。那么该选哪一台？

然后我被叫到老板办公室，被开除了。
为什么？因为如果我考虑每个事务所需的时间，就会发现，在A上做一次程序1的事务和一次程序2的事务，总共需要0.5秒。而在B上，它需要9/14秒。

9/14秒比0.5秒长，所以其实A更好，而不是B。
那么为什么我会得出错误的结论？这是因为用算术平均值来处理速率是没有意义的。

算术平均值适用于你希望数字越小越好的情况，但对于速率，你希望数字越大越好。所以使用算术平均值会产生误导。
那么我们该怎么做呢？如果我要将其转换回速率，我应该将总时间除以事务的数量。

这样我就可以看到，A实际上比B更好。
为什么我要在计算机架构课上讲这个？因为这是人们过去在衡量计算机性能时的一个根本性错误。他们平均了速率，但速率不能这样平均。

这种方法有点像电阻的串联和并联。在并联电阻中，是最慢的那一个决定了平均电阻的大小。同样，在这里，我需要强调哪一个是较小的数字，因为它是决定因素。

这是一个非常重要的概念，似乎有点基础性，但却很有用。

如何综合这些概念

如果你喜欢较小的数值结果，或者较大的数值结果，首先要把数据转换成时间。然后，在计算完成后，将它们转换回速率。这个方法的数学公式是：

η=11−f+fS\eta = \frac{1}{1 - f + \frac{f}{S}} η=1−f+Sf​1​

这个公式很重要，堪称经典的“Amdahl定律”。

Amdahl定律的背景

在1967年的秋季联合计算机会议上（当时一年只有两场关于计算机的大会），有位名叫Gene Amdahl的人发表了一篇演讲。他提到，当时有一些人声称单台计算机的性能已经达到极限，只有通过多台计算机的互连协作才能取得真正的进步。换句话说，他在讨论并行计算的局限性。

Amdahl指出，并行计算的瓶颈在于顺序部分（sequential part），因为它不能被并行化。无论你如何加速并行部分，瓶颈仍然存在。

Amdahl定律的解释

假设程序运行时间可以分为三个部分：A、B和C。现在我们发明了一种“神器”，可以加速B部分的运行时间。那么即使B部分的运行时间被缩短到接近0，A和C的时间仍然是决定性的。这意味着，即使我们让B的速度无限提高，程序的整体速度提升仍然受到限制。

公式可以表示为：

Speedup=1(1−f)+fS\text{Speedup} = \frac{1}{(1 - f) + \frac{f}{S}} Speedup=(1−f)+Sf​1​

其中，fff 是可以加速的部分占总时间的比例，SSS 是加速比。

一个实际例子

假设我在一家设计飞机机翼的公司工作，用我的电脑模拟喷气机需要一周的时间。一天，我在邮件室里看到一张宣传单，上面写着一种名为“Acme Hyperbole”的新电脑，配备了10万个处理器。我的电脑用一个处理器需要一周，那么这台新电脑应该只需要6秒。

但实际上，我检查了代码后发现，模拟中只有80%的部分可以被并行化，另外20%是顺序部分，比如分支、依赖关系等。

用Amdahl定律计算：

Speedup=1(1−0.8)+0.8100000≈5\text{Speedup} = \frac{1}{(1 - 0.8) + \frac{0.8}{100000}} \approx 5 Speedup=(1−0.8)+1000000.8​1​≈5

也就是说，即使使用这台超级电脑，模拟时间也只减少到原来的五分之一，这与预期的6秒相差甚远。这显然不值花费一大笔钱去购买。

总结

并行计算的提升受到顺序部分的限制。无论新技术多么先进，如果顺序部分占主导地位，性能的提升都会受到限制。这就是Amdahl定律所揭示的核心思想。

Gustafson 的观点

他提出了一个设想：假设你有一个并行程序，而当前的并行程序运行时间为 T1+T2T_1 + T_2T1​+T2​。

  • T1T_1T1​ 是顺序部分（sequential part）。
  • T2T_2T2​ 是并行部分（parallel part）。

他的观点很简单：如果你给我更多的计算能力，我不会缩小并行部分，而是会增加问题的规模，增加计算的精度，做更多的工作。我不会让那些处理器核心闲置，而是会让问题变得更复杂。

于是，这个并行程序的顺序时间就是 T1+T2T_1 + T_2T1​+T2​，对于 nnn 个处理器，其加速比（Speedup）是：

Speedup(n)=T1+T2T1+T2n\text{Speedup}(n) = \frac{T_1 + T_2}{T_1 + \frac{T_2}{n}} Speedup(n)=T1​+nT2​​T1​+T2​​

Gustafson 定律

如果进行数学推导，并定义一个新的变量 α\alphaα，即原始程序的顺序部分的比例：

α=T1T1+T2\alpha = \frac{T_1}{T_1 + T_2} α=T1​+T2​T1​​

那么加速比可以表示为：

Speedup=α+(1−α)⋅n\text{Speedup} = \alpha + (1 - \alpha) \cdot n Speedup=α+(1−α)⋅n

但这实际上是一个“谎言”。Gustafson 改变了问题的规模和基准。他改变了基线（baseline），这让加速比看起来好得多。他的观点是：如果我改变程序，把那些处理器用在其他地方，我的加速比就会显得很高。

弱扩展（Weak Scaling）和强扩展（Strong Scaling）

这种方法被称为“弱扩展”（Weak Scaling），在超级计算会议上，几乎所有的论文都会展示弱扩展的结果。在这些论文中，绘制的每个点实际上对应一个不同的问题规模。他们“欺骗”了你，但他们别无选择。因为如果问题规模不变，根据 Amdahl 定律，性能提升会受到很大的限制。Amdahl 的方法被称为“强扩展”（Strong Scaling），它是一个很好的概念，但难以实现。

作为架构师，我们无法改变算法，也无法改变程序逻辑，所以我们通常忽略 Gustafson 定律。
但如果你是一个国家实验室，并且需要为数十亿美元的纳税人资金正名，那么弱扩展就非常有吸引力了。
